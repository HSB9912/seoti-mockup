즉시 필요

현재 admin-business.html에 9개 매장의 사업자번호·담당자·이메일·주소·정산금액이 공개 GitHub Pages에 노출 중. 개인정보보호법 위반 가능

WHY제3자(입점처)의 개인정보를 동의 없이 공개 시 처벌 대상. 매장 정산액·수수료는 영업 비밀.
법
개인정보 처리방침 작성·게시

회원가입 시 "동의" 텍스트만 있고 실제 문서 X. 개인정보보호법 30조는 처리방침 공개를 의무화함.

WHY수집 항목·이용 목적·보관 기간·파기 절차 명시 필수. 미이행 시 과태료 1,000만 원 이하.
법
이용약관 작성·게시

전자상거래 운영 시 약관 게시 의무. 환불·배송·교환 정책 포함.

법
통신판매업 신고

온라인 판매 시 관할 지자체에 신고 필수. 신고번호 사이트 푸터에 표시 의무.

WHY「전자상거래법」 12조. 미신고 시 과태료 3,000만 원 이하.
법
사업자 정보 사이트 푸터 표시

상호·대표자명·사업자등록번호·통신판매업번호·주소·연락처·이메일 명시.

법
만 14세 미만 회원가입 처리

법정대리인 동의 절차 필요. 개인정보보호법 22조의2. 또는 14세 이상만 가입 가능하게 차단.

법
회원 동의 항목 분리 (필수/선택)

이용약관·개인정보처리방침은 필수, 마케팅 수신은 선택. 각각 개별 동의 체크박스로.

WHY일괄 동의 금지. 「개인정보보호법」 22조.
법
환불·교환 정책 페이지

구매 후 7일 이내 청약철회 등 법정 권리 명시. QnA 페이지나 별도 페이지로.

법

🔒보안 · 인증

중요
비밀번호 해싱 (bcrypt/Argon2)

Supabase Auth는 기본 제공. 직접 구현 시 평문 저장 절대 금지.

WHY유출 시 평문 비밀번호 노출은 손해배상 폭발. 「개보법」 위반.
보안
Supabase Row Level Security (RLS) 설정

모든 테이블에 RLS 활성화 + policy 작성. auth.uid()로 본인 데이터만 접근.

WHY기본값은 anon key로 누구나 모든 데이터 읽기 가능. 반드시 RLS 켜야 함.
보안
관리자 권한 분리 (admin role)

Supabase auth.users에 role 필드. admin 전용 테이블은 role=admin만 SELECT 가능하게 RLS.

현재 admin-guard.js는 JS 비밀번호라 누구나 우회 가능 — 반드시 교체.

보안
API 키 환경변수 분리 (.env)

Supabase URL/anon key는 환경변수로. Service role key는 절대 브라우저 노출 금지 (서버에서만).

WHYservice_role 키 노출 시 RLS 우회 가능 → DB 전체 탈취 위험.
보안
이메일 인증 활성화

Supabase Auth → Email confirmations ON. 가짜 이메일 가입 방지.

보안
로그인 시도 횟수 제한 (rate limit)

Supabase 기본 제공 + 추가로 IP별 제한 고려. 무차별 대입 공격(brute force) 방어.

보안
HTTPS 강제 (Cloudflare)

Cloudflare → SSL/TLS → Full(strict). HTTP 접근 시 자동 HTTPS 리다이렉트.

보안
CSRF·XSS 방어

Supabase JS 클라이언트는 기본 토큰 처리. 사용자 입력 출력 시 textContent 사용 (innerHTML X).

보안
이미지 업로드 검증 (Cloudflare Images)

파일 타입·크기 서버에서 재검증. 악성 파일(.exe 등 이미지 위장) 차단. EXIF 메타데이터 자동 제거.

보안

🗂데이터 모델 (Supabase 스키마)

중요
핵심 테이블 설계

필요한 테이블 (예시): users · profiles · products · product_images · orders · order_items · cart_items · wishlists · qna · qna_replies · vendors · settlements · expenses · archive_photos · site_content · notices

DB
상태값(enum) 정의

제품 상태: in_stock · sold_out · discontinued
주문: pending · paid · shipping · delivered · refunded
정산: 완료 · 확인필요 · 미입금

DB
인덱스 설계

자주 조회되는 컬럼에 index: products(category, state), orders(user_id, created_at desc), qna(category, status) 등

DB
외래키 관계 + ON DELETE 정책

주문자(user)가 탈퇴 시 주문은 어떻게? CASCADE? SET NULL? 보관 의무 5년 (전자상거래법).

DB
created_at·updated_at 자동 기록

모든 테이블에 타임스탬프. updated_at은 트리거로 자동 갱신.

DB
이미지 URL 저장 방식

Cloudflare image ID만 DB에 저장 (cf_image_id). delivery URL은 코드에서 생성. 변경 유연성 확보.

DB
가격 통화 단위 고정

원화 정수만 사용 (소수점 X). price INTEGER NOT NULL

DB

⚙️운영 · 개발 환경

필요
스테이징 환경 분리

Supabase 프로젝트를 dev/prod 둘로. 실수로 운영 DB 망가지는 거 방지.

운영
백업 정책

Supabase Pro 플랜은 7일 자동 백업. Free는 수동. 정기적으로 SQL dump 받기.

WHY실수로 삭제·해킹·과실로 데이터 잃으면 복구 불가.
운영
로그·모니터링

Supabase logs 정기 확인. 에러 발생 시 알림 받을 채널 (이메일·Discord) 설정.

운영
이메일 발송 설정

주문 확인·재입고 알림·QnA 답변 등. Supabase Auth는 기본 발송 가능, 추가 발송은 Resend·Mailgun 등 연동.

운영
도메인 연결

현재 3eotty-afk.github.io/seoti-mockup. 실제 운영 시 자체 도메인 (예: seoti.studio) → Cloudflare DNS → 호스팅 연결.

운영
분석 도구

Google Analytics 또는 Plausible/Umami. 어떤 페이지가 인기 있는지·이탈 지점 파악.

운영
SEO 메타 태그

각 페이지에 <meta description>, OG 태그 (카톡·트위터 공유 미리보기).

운영

📦기존 데이터 마이그레이션

중요
입점처 9개 → vendors 테이블

현재 admin-business.html에 JS 배열로 들어있는 VENDORS 9개를 CSV 또는 SQL insert로 옮기기.

마이그
정산 데이터 33건 → settlements

월별·입점처별 정산. settlement_date·paid_date 등 날짜 컬럼 표준화.

마이그
세금·관리비 → expenses

노션 페이지의 6건. 분류(사무실/세금) enum화.

마이그
작가 컬렉션 → products 테이블 등록

모든 이미지를 Cloudflare R2 (stty-images) 에 업로드 후 image_id 저장.

마이그
Archive 사진 → archive_photos

다꾸 샘플·작업 사진 기존 자료들을 일괄 업로드. 분류·날짜·메모 메타데이터 포함.

마이그

💳결제 연동

중요
결제 PG사 선택·계약

토스페이먼츠 · 포트원(아임포트) · KG이니시스 등. 수수료·정산 주기·약관 비교.

결제
전자결제 표준 준수

본인인증·결제 도중 정보 보호·결제 결과 검증. PG사 SDK 가이드 따르기.
